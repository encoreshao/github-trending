# 404 Page — Design

Date: 2026-07-04

## Problem

The site has no handling for unmatched routes. Inside the React Router SPA, navigating to an unknown path renders a blank page (no route matches). Separately, the production nginx config (`tmp/nginx.conf`) has no SPA fallback, so a direct browser hit or refresh on *any* path — even a valid one like `/weekly` — returns nginx's own bare 404 without ever reaching the React app. Both need fixing for a working, styled 404 experience.

## Approach

Add a catch-all React Router route rendering a new `NotFoundPage`, styled consistently with the rest of the site (dark/glassmorphism, Header/Footer) and themed as a GitHub API 404 joke — reusing the homepage's existing `.code-window` mockup treatment rather than inventing a new visual pattern. Separately, add the missing SPA fallback to `tmp/nginx.conf` so nginx routes unmatched paths to `index.html`, letting the client-side router (and therefore `NotFoundPage`) actually run for direct hits and refreshes.

## Components

**`src/pages/NotFoundPage.jsx` (new)**
- Renders `Header`, a main content area, and `Footer` — same shell pattern as `HomePage.jsx`/`DemoPage.jsx`.
- Main content: a "code window" mockup (reusing the homepage's `.code-window`/`.window-header`/`.window-controls`/`.code-content` visual structure — window chrome with red/yellow/green dots, a title bar, and a `<pre><code>` block) showing a fake GitHub API 404 JSON response:
  ```json
  {
    "message": "Not Found",
    "documentation_url": "https://github.ranbot.online/",
    "status": "404"
  }
  ```
- Below the code window: a large "404" numeral, a short GitHub-flavored tagline ("This repo doesn't exist... yet."), a primary "Back to Home" button (navigates to `/`), and three smaller secondary links to `/weekly`, `/monthly`, and `/demo`.
- No data fetching — this is a fully static page.

**`src/pages/NotFoundPage.css` (new)**
- Follows the established palette: navy `#0f172a` background, blue accent `#60a5fa`/`#3b82f6`, muted text `#94a3b8`, card surfaces `rgba(30, 41, 59, 0.5)`.
- Reuses the same window-chrome visual proportions as the homepage's `.code-window` (not a shared CSS file — the homepage's version is scoped inside `HomePage.css` and tightly coupled to the hero's two-column grid; this page's version is a simpler, centered, standalone treatment, so it gets its own small CSS block rather than trying to extract a shared component for a single reuse).
- Responsive at the site's existing 968px/640px breakpoints: code window shrinks, button row stacks on mobile.

**`src/NewApp.jsx` (modify)**
- Add `import NotFoundPage from './pages/NotFoundPage';` and `<Route path="*" element={<NotFoundPage />} />` as the last route in the `<Routes>` block (route order matters in React Router — the catch-all must come after all specific paths).

**`tmp/nginx.conf` (modify)**
- Add a `location / { try_files $uri $uri/ /index.html; }` block inside the HTTPS (`listen 443 ssl`) server block, alongside the existing `root`/`index` directives. This file is gitignored (per `.gitignore`'s `tmp/` entry) and is a deployment reference the user applies manually on the server — this change updates the reference copy in the repo but does not itself touch the live server.

## Data flow

None — this is a static page with no network calls, matching the "no data fetching" nature of a 404 route.

## Error handling

N/A — there's no failure mode within the page itself; it *is* the failure-mode handler for the rest of the app.

## Testing

No test framework exists in this repo. Verification: `npm run build`, then in the dev server, navigate to an arbitrary unmatched path (e.g. `/this-does-not-exist`) and confirm `NotFoundPage` renders with working nav links; confirm all existing routes (`/`, `/weekly`, `/monthly`, `/demo`, `/subscribe`) still resolve to their own pages, not the catch-all. The nginx change can't be verified locally (no local nginx in this dev workflow) — it's a config-only change reviewed by inspection, applied and verified by the user on the actual server.

## Out of scope

- Server-side rendering or a real HTTP 404 status code for the SPA route (React Router SPAs serve everything as 200 from the static host; nginx's `try_files` fallback means even genuinely missing paths return `index.html` with a 200 — this is normal, expected behavior for a client-side-routed SPA, not a bug to fix here).
- Actually deploying/restarting nginx on the production server — the user applies the config change themselves.
- Any other site-wide navigation restructuring.
