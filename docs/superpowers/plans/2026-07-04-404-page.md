# 404 Page Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a styled, on-brand 404 page for unmatched routes, and fix the production nginx config so direct URL hits/refreshes actually reach it.

**Architecture:** A new `NotFoundPage` component (Header/Footer shell + a fake GitHub-API-404 "code window" mockup + CTAs) is wired as React Router's catch-all route. Separately, `tmp/nginx.conf` gets a `try_files` SPA fallback so nginx serves `index.html` for any unmatched path instead of its own bare 404.

**Tech Stack:** React 18, react-router-dom. No new dependencies.

## Global Constraints

- Follow the existing dark/glassmorphism palette: navy `#0f172a`, card surface `rgba(30, 41, 59, 0.5)`/`rgba(30, 41, 59, 0.8)`, blue accent `#60a5fa`/`#3b82f6`, muted text `#94a3b8`.
- `NotFoundPage` gets its own self-namespaced CSS classes (`.notfound-*`) rather than reusing `HomePage.css`'s `.code-window`/`.button` classes — those are plain global CSS classes scoped only by convention, and depending on `HomePage.css` having already loaded (via routing order) would be fragile. Match their visual proportions and colors, but don't share the class names or the file.
- The catch-all route (`path="*"`) must be the LAST route in `NewApp.jsx`'s `<Routes>` — React Router matches in order, so anything after it would never be reached, and if it were placed earlier it would shadow real routes.
- `tmp/nginx.conf` is gitignored (per `.gitignore`'s `tmp/` entry) and is a deployment reference file the user applies manually on their server — this task edits the reference copy in the repo; it does not touch the live server.
- No test framework exists in this repo; verification is `npm run build` plus manual dev-server checks for the frontend task. The nginx change is verified by inspection only (no local nginx in this workflow).

### Task 2: Add the nginx SPA fallback

**Files:**
- Modify: `tmp/nginx.conf`

**Interfaces:** None — this is a standalone config file, not consumed by any other task.

- [ ] **Step 1: Add the `try_files` fallback to the HTTPS server block**

Current (the HTTPS server block in `tmp/nginx.conf`):

```nginx
server {
	listen 443 ssl;
	server_name github.ranbot.online;

	ssl_certificate /etc/letsencrypt/live/github.ranbot.online/fullchain.pem;
	ssl_certificate_key /etc/letsencrypt/live/github.ranbot.online/privkey.pem;
	ssl_trusted_certificate /etc/letsencrypt/live/github.ranbot.online/chain.pem;

	include snippets/ssl.config;
	include snippets/letsencrypt.conf;

  root /var/www/production/github-trending/current/public;
  index index.html;
}
```

New:

```nginx
server {
	listen 443 ssl;
	server_name github.ranbot.online;

	ssl_certificate /etc/letsencrypt/live/github.ranbot.online/fullchain.pem;
	ssl_certificate_key /etc/letsencrypt/live/github.ranbot.online/privkey.pem;
	ssl_trusted_certificate /etc/letsencrypt/live/github.ranbot.online/chain.pem;

	include snippets/ssl.config;
	include snippets/letsencrypt.conf;

  root /var/www/production/github-trending/current/public;
  index index.html;

  location / {
    try_files $uri $uri/ /index.html;
  }
}
```

Leave the HTTP (`listen 80`) server block untouched — it only issues the `return 301 https://$host$request_uri;` redirect and never serves static files, so it needs no fallback.

- [ ] **Step 2: Verify by inspection**

There's no local nginx in this workflow to test against. Confirm by reading the file: the `location /` block is inside the `listen 443 ssl` server block (not the `listen 80` block), and `try_files $uri $uri/ /index.html;` is syntactically valid nginx (semicolon-terminated, matches standard SPA-fallback syntax).
